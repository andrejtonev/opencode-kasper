1. Overview & Goals

The Observer Plugin is an opencode plugin that monitors all agent sessions (main + subagents), evaluates how well each agent follows its instructions, and iteratively improves behavior by updating AGENTS.md and agent-specific prompt files.

Core Goals
  • Observe — passively listen to all session events, tool calls, and messages across main agent and subagent child sessions
  • Evaluate — score each session against the user's stated goals/instructions via LLM-as-judge
  • Improve — suggest and apply refinements to AGENTS.md (project-level rules) and individual agent prompt files (.opencode/agents/{name}.md or opencode.json)
  • Backup — maintain version history of all prompt changes for rollback
  • Cross-session — aggregate observations across all sessions, including per-agent breakdowns
  • User control — command-driven interaction: trigger evaluations manually, approve/reject improvements, review history

3. Plugin API Surface

The plugin exports a server-side plugin function. There is one plugin instance per opencode instance — all state is shared across all sessions (main + subagent child sessions). The plugin receives { project, directory, client } at init and returns a hooks object.

3.1 Server Plugin Hooks

HookStatusPurpose
eventDONEListen to all session lifecycle events — scoring is triggered by polling (setInterval via pollAndEvaluate()) rather than idle events
chat.messageDONECapture user instructions (creates PendingEval) and assistant responses (appends to PendingEval)
tool.execute.afterDONECapture tool calls made by the agent — tool name, args, and actual results recorded
tool.execute.beforeDONECapture tool inputs before execution — used for slash command interception via command.execute.before
experimental.session.compactingDONEInject top weaknesses + average score + per-agent stats + score emoji + trend arrows into compaction context
experimental.compaction.autocontinueTODOOptionally suppress auto-continue while observer is running an evaluation
session.createdDONEDetect new subagent child sessions — learn agent type/name for per-agent tracking, populate agentSessionIDs
session.deletedDONEClean up pending evaluations, agent registry, idle times, and parent/child mappings for deleted sessions
Key Architectural Insight

The plugin function runs once when opencode loads, and the hooks object is used for ALL sessions — both primary agent sessions and subagent child sessions. The pendingEvaluations Map is keyed by sessionID, so subagent sessions naturally get their own entries. All state (stateStore, agentsMd, scorer) is shared in the closure across all sessions.

3.2 Custom Tools

The plugin registers 9 custom tools for the main agent to interact with the observer:

observer_status        — Ask the observer for its current evaluation score and aggregate stats
observer_improve       — Request that the observer suggest AGENTS.md improvements
observer_history       — Retrieve adherence history and scores
observer_auto          — Enable/disable automatic AGENTS.md updates for the session
observer_score_agent   — View the most recent evaluation for a specific agent type
observer_reject        — Reject an improvement suggestion by pattern or index
observer_rejections    — List all rejected weakness patterns
observer_unreject      — Remove a rejection, restoring the pattern
observer_steer         — Provide evaluation guidance that influences the scorer

  • Suggested improvements with reject/apply controls

4. Adherence Scoring Engine

4.1 Scoring Model

The scorer uses an LLM call via a dedicated child session with structured output (json_schema). The child session is created, prompted, and immediately deleted — keeping the scoring prompt out of the main session's context.

ScoreCard {
  session_id: string;
  message_id?: string;
  timestamp: number;
  overall_score: number;          // 0.0 - 1.0
  categories: {
    instruction_following: number;  // Did the agent do what was asked?
    completeness: number;           // Did it finish the task?
    proactiveness: number;          // Did it over-step or under-step?
    code_quality: number;           // Quality of generated code
    communication: number;          // Clarity of explanations
  };
  strengths: string[];
  weaknesses: string[];
  suggested_agents_md_update?: string;      // A suggested AGENTS.md addition
  suggested_agent_prompt_update?: string;    // A suggested agent prompt addition
}

On scoring failure, defaults to 0.5 for all categories with weaknesses: ["Scoring failed: ..."]. Scoring sessions are always deleted in a finally block.

4.2 Evaluation Trigger Flow

The judging side-session is triggered automatically by a polling timer (setInterval every 10s via pollAndEvaluate()):

  1. chat.message (user role) → creates a PendingEval entry in the Map, keyed by sessionID
  2. chat.message (assistant role) → appends response text parts to the pending eval
  3. tool.execute.after → appends tool call records (currently with result: "(see output)" — actual outputs discarded)
  4. setInterval tick → pollAndEvaluate() → calls runEvaluation() which:
    • Reads current AGENTS.md content
    • Builds ScorerInput from pending data
    • Calls scorer.evaluate() → creates child session, prompts with structured output, deletes child session
    • Records the session + score card in state store
    • If score < threshold: calls considerImprovement()

Code location: Trigger at src/index.ts:115-142, child session creation at src/scorer.ts:103-120.

4.3 SCORE_PROMPT

Defined at src/scorer.ts:16-26. The eval LLM is asked to score on 5 dimensions, identify strengths/weaknesses, and optionally suggest AGENTS.md additions. The full prompt (assembled at src/scorer.ts:201-228 in buildEvalPrompt()) appends:

  • User's original instruction
  • Agent's response text
  • Tool calls made (args shown, results truncated to 500 chars)
  • Current AGENTS.md content
  • Current agent prompt (if available)
  • User steering guidance (if set via /observer steer)
Model Configuration

The config.model field (default: "opencode/deepseek-v4-flash-free") is parsed via parseModelString() and passed as model option to both sessionClient.create() and sessionClient.prompt().

4.5 Scoring Detail Level

The config.detail_level field controls what context is included in the scoring prompt, balancing depth against token cost:

LevelTool call detailAGENTS.mdAgent promptResult truncation
minimalTool names onlyExcluded (noted as present)Excluded (noted as present)N/A
standard (default)Args + 500-char resultsIncludedIncluded500 chars
thoroughArgs + 2000-char resultsIncludedIncluded2,000 chars

4.4 Triggers (Design vs Implementation)

  • After each tool.execute.after — NOT implemented; only accumulates data
  • After each complete assistant response — NOT implemented; only accumulates data
  • Poll-based via setInterval — IMPLEMENTED (primary trigger, 10s pollAndEvaluate())
  • On session.compacted — NOT implemented

Currently evaluates only once per session (at poll time, with re-evaluation on new messages). Mid-session evaluation and per-response scoring are design goals but not built.

5. Subagent Observation & Prompt Updates

5.1 OpenCode Subagent Model

OpenCode has two agent modes:

  • Primary agents (mode: "primary"): Build, Plan, Compaction, Title, Summary — tab-switchable, main conversation surface
  • Subagents (mode: "subagent"): General, Explore, Scout, + custom — invoked by the primary agent via the Task tool, or manually by @ mentioning

Subagents create child sessions navigable with keybinds. Each agent has its own system prompt configured in one of two ways:

  1. opencode.json: agent.{name}.prompt: "{file:./prompts/build.txt}" — file path reference
  2. Markdown files: .opencode/agents/{name}.md (project) or ~/.config/opencode/agents/{name}.md (global) — frontmatter (mode, description, model, permission, etc.) + body is the system prompt

5.2 How Subagents Are Observed

The existing plugin hooks (chat.message, tool.execute.after, event) receive sessionID and fire for ALL sessions — including subagent child sessions. Since the pendingEvaluations Map is keyed by sessionID, subagent sessions naturally accumulate their own pending evals. What's missing is:

  1. Session type detection — hook session.created to learn which agent type/name spawned each session
  2. Per-agent tracking — extend SessionRecord with agent_name and agent_type fields
  3. Per-agent aggregation — compute separate weakness/strength stats per agent (General's weaknesses differ from Build's)
  4. Agent prompt context — include the agent's current prompt in scoring so the eval LLM can suggest agent-specific improvements

5.3 Agent Prompt Manager (New Component)

A new AgentPromptManager class (paralleling AgentsMdManager) would:

  • Detect where an agent's prompt lives: markdown file vs opencode.json
  • Read the current prompt (for inclusion in scoring context)
  • Write improvements — for markdown agents, inject a new section in the body; for JSON-based agents, add/replace section in the prompt text
  • Update description — subagent description frontmatter controls when the primary agent invokes it; if the observer detects a capability gap, it should update the description too
  • Backup before modification, same timestamped scheme as AGENTS.md

5.5 Design Decisions to Resolve

  • JSON injection: Modifying agent prompts in opencode.json is messy — should we instead create a companion markdown agent file that overrides/supplements?
  • New agent creation: If a subagent has no markdown file yet, should the observer create one (preferred) or modify opencode.json?
  • Description updates: When observer detects that a subagent's description doesn't capture its actual capabilities, should it update the description frontmatter so the primary agent knows when to invoke it?
  • Task permission interaction: The permission.task config controls which agents can invoke which subagents. The observer should not override these — just improve the prompts.

6. AGENTS.md Management

6.1 Update Strategy

When the observer detects a pattern of non-adherence (e.g., "agent keeps using wrong test framework"), it formulates an AGENTS.md update. The strategy:

  1. Accumulate — collect weakness signals across multiple sessions
  2. Threshold — only suggest updates after N observations (configurable, default 3)
  3. Section injection — insert/replace a named ## Section in AGENTS.md
  4. User approval — ask the user before modifying AGENTS.md (configurable to auto-approve) — DESIGNED but NOT IMPLEMENTED
  5. Auto-apply — on approval, backup current AGENTS.md, write new version
Current Behavior

The implementation respects the auto_update_agents_md config flag. When false (default), improvements queue as pending and await /observer apply. When true, or when session auto-update is toggled on via /observer auto on, improvements auto-apply. Weakness matching uses fuzzy similarity (exact + substring + Jaccard word overlap via weaknessSimilarity()). Rejected patterns are tracked with fuzzy matching and never re-suggested.

6.2 Sections Observer Manages

The observer maintains a dedicated section in AGENTS.md:

## Observer Inferred Instructions
*This section is managed by the Observer plugin. Do not edit manually.*
- Use `vitest` for unit tests (not Jest)
- Always run `tsc --noEmit` after edits
- Prefer `path.join()` over string concatenation
...

It may also update other sections when confidence is high (e.g., updating ## Code Standards).

  • Multiple rollbacks revert further back in the backup chain
  • Backup index stored in state store for easy navigation
  • DONE for both AGENTS.md and agent prompts

8. Cross-Process Safety

8.1 Locking Strategy

When multiple opencode instances run on the same project, concurrent writes to state.json, AGENTS.md, and agent prompt files can cause data loss. The observer now uses cross-process file locking:

  • Lock acquisition: acquireLock(path, timeoutMs) in src/lock.ts — exclusive file creation via fs.open(path, 'wx') (atomic on all platforms)
  • Stale detection: Locks older than 30 seconds are considered stale and automatically cleaned up — prevents deadlock from crashed processes
  • Retry with backoff: Exponential backoff (10ms → 20ms → ... → 500ms) up to 5-second default timeout
  • PID tracking: Lock files contain the owning process PID for debugging

8.2 State Merge-Before-Write

For state.json, the flush acquires the lock, then re-reads the current file to merge in any sessions/improvements/rejections written by other processes since our last load. Only then does it write. This prevents the classic read-modify-write race.

8.3 Protected Resources

state.jsonstate.json.lockLock → merge external → write → unlock
AGENTS.mdAGENTS.md.lockLock → read → modify → atomic write → unlock
Agent prompts{agent}.md.lockLock → read → inject section → atomic write → unlock

9. User Interaction Model

9.1 Slash Commands

Thirteen slash commands implemented under /observer:

CommandDescription
/observer status [agent]Show aggregate stats, top weaknesses, recent sessions, ASCII sparkline trend, improvement deltas
/observer score [agent|sessionID]Trigger manual evaluation, view agent-specific results, or evaluate a historical session by ID
/observer suggest [agent]Generate improvement suggestions (AGENTS.md + agent prompts), excluding rejected patterns
/observer applyApply the next pending improvement to AGENTS.md or agent prompt

Commands are intercepted via command.execute.before hook. Output is set on output.message with output.stop = true to prevent passthrough.

Additionally, 9 custom tools are registered (see §3.2) allowing the agent to query and control the observer programmatically.

9.3 Toast Notifications

The plugin uses showToast() to notify users non-intrusively:

  • Medium/high score (>=0.4): info toast with score emoji (green/yellow/red) and percentage
  • Improvement pending: "Improvement suggestion available. /observer apply to review."
  • Improvement applied: "AGENTS.md updated with new instructions."

10.2 Per-Agent Aggregation

Per-agent breakdowns are fully implemented via getAgentAggregate(agentName) and getAgentSessions(agentName). Each agent's sessions are tracked separately:

  • Build agent weaknesses tracked separately from Explore agent weaknesses
  • Improvements target the right prompt file (AGENTS.md for project-wide issues, agent prompt for agent-specific issues)
  • Agent session IDs tracked via agentSessionIDs map, populated on session.created
  • Rejected patterns persisted in state.json under rejected_patterns array

10.3 Single Instance / Cross-Project

The observer is project-scoped (per .opencode/ directory). The plugin context is shared across all sessions in that project instance — the pendingEvaluations Map, stateStore, and agentsMd are all created once in the plugin closure (src/index.ts:25-53) and reused for every session.

13. Open Questions

  • Scoring latency: LLM-as-judge calls take time. Should scoring be async (fire-and-forget) or block? Async seems safer but we lose real-time feedback.
  • Scoring cost: Each evaluation costs tokens. How frequently should we evaluate? Every message, or only at session end?
  • Cross-session identity: How does the observer correlate that two sessions belong to the same "task"? Via user session title? Manual tagging?
  • Conflict resolution: RESOLVED — The rejection system tracks user-rejected patterns with fuzzy matching and persists them in state.json. Rejected patterns are never re-suggested. Users can unreject via /observer unreject.
  • Agent prompt vs AGENTS.md: RESOLVED — The observer updates BOTH: AGENTS.md for project-wide patterns, and individual agent prompt files for agent-specific issues. See §5 for design.
  • Agent prompt modification format: When an agent's prompt lives in opencode.json (as a {file:...} reference), should the observer inject improvements into the referenced file, or create a companion .opencode/agents/{name}.md file?
  • Integration with /init: The /init command also modifies AGENTS.md. How do we reconcile observer updates with /init?
  • Plugin load order: If multiple plugins hook experimental.session.compacting, our context injection must compose with theirs.
  • Desktop vs TUI vs CLI: The plugin API is the same across all runtimes, but the TUI plugin can provide richer UI (panels, dialogs). Should we support both?
  • How do we inject updated rules into a running session? AGENTS.md and agent prompts are read at session start and after compaction. We could use tui.prompt.append or inject via experimental.session.compacting. The compaction hook is our key injection point.
  • Scoring model passthrough: How exactly do we specify a model when creating a child session? Need to check SDK API for model override in session.create() or session.prompt().
  • Subagent session identity: How does session.created tell us which agent type/name spawned the session? Need to inspect the event shape — is there an agent or agent_id field?
